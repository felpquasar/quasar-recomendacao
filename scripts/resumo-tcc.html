1. Objetivo do Projeto

Desenvolver um sistema de recomendação de produtos B2B para a Quasar Barber, distribuidora de produtos para barbearias em Codó-MA. O sistema analisa o histórico de vendas para identificar padrões de compra e sugerir automaticamente produtos complementares no momento do pedido.

O trabalho aplica o algoritmo de Association Rules (Market Basket Analysis) — técnica consolidada em Data Mining — adaptada ao contexto B2B.

Problema resolvido

  • Vendedores não sabem quais produtos sugerir em cada pedido
  • Oportunidades de cross-sell perdidas por falta de análise de dados
  • Decisões baseadas em intuição, não em histórico quantitativo

Solução entregue

  • API REST: recebe produto → retorna top 3 produtos comprados junto com métricas
  • Dashboard React funcional com KPIs, busca interativa e cards de recomendação
  • Algoritmo treinado com dados reais de vendas (Pedidos_2026.csv importado)
  • Validação do algoritmo com Precision, Recall e F1-Score
Exemplo com dados reais: Vendedor informa "BALM 120ML - FOX FOR MEN" → sistema retorna "SHAMPOO PARA BARBA 120ML - FOX FOR MEN" (100% de confiança, Lift 16.5) e "OLEO PARA BARBA SPRAY 60ML - FOX FOR MEN" (100% de confiança, Lift 16.5).

6. Algoritmo de Recomendação — Association Rules

As três métricas

MétricaFórmulaSignificado prático
Supportfreq(A ∩ B) / total_pedidosQuão comum é o par A+B no dataset. Filtra raridades (threshold: 2%).
Confidencefreq(A ∩ B) / freq(A)P(B|A): dado que A foi comprado, prob. de B também. Métrica principal (threshold: 30%).
Liftconfidence(A→B) / P(B)Lift > 1 = correlação real. Lift = 1 = independência. Lift 16.5 = 16× mais chance.

Thresholds configurados (src/config/constants.js)

ParâmetroValorJustificativa
MIN_CONFIDENCE0.30 (30%)Rejeita regras fracas — recomendação precisa ter pelo menos 30% de chance
MIN_SUPPORT0.02 (2%)Filtra pares raros — par precisa aparecer em ≥2% das transações
TOP_N3Máximo de 3 recomendações por produto no frontend

Fluxo do algoritmo — O(T × P²)

// T = transações, P = produtos únicos // Dataset atual: 33 × 33² = ~36.000 operações (<50ms) Para cada produto A: para cada produto B (onde B ≠ A): freq_A = pedidos onde A aparece freq_AB = pedidos onde A e B aparecem juntos confidence = freq_AB / freq_A support = freq_AB / total_pedidos lift = confidence / (freq_B / total) se confidence >= 0.30 e support >= 0.02: adiciona regra {A → B, confidence, support, lift} // Ordena por confidence desc → retorna top 3 para o produto consultado

9. Cache Redis

Estratégia cache-aside

  • 1. Verifica Redis pela chave → se hit, retorna imediatamente com source: "cache"
  • 2. Se miss: busca Supabase → calcula algoritmo → armazena resultado no Redis
  • 3. Próxima chamada ao mesmo produto: resposta em <10ms

Degradação graceful (services/cache.js)

Se Redis estiver offline, todas as funções retornam silenciosamente: get → null, set → void. API continua funcionando normalmente.

10. Validação do Algoritmo — Precision, Recall, F1

Implementado na Semana 5 via services/validacao.js e routes/validacao.js. Mede a qualidade do algoritmo com dados reais.

Metodologia — Train/Test Split 80/20

Dataset completo (33 transações) │ ├── 80% → Treino (26 transações) → calculateAssociation() → regras └── 20% → Teste (7 transações) → validar se regras acertam Para cada transação de teste (com ≥ 2 itens): - primeiro produto = antecedente (input) - demais produtos = ground truth (o que o cliente realmente comprou) - busca regras com aquele antecedente - compara recomendações com ground truth → calcula P, R, F1

Fórmulas das métricas

MétricaFórmulaPergunta respondida
Precisionacertos / total_recomendacoesDas recomendações feitas, quantas o cliente comprou?
Recallacertos / total_compras_reaisDas compras realizadas, quantas foram recomendadas?
F1-Score2 × (P × R) / (P + R)Média harmônica — penaliza P ou R muito baixo

Endpoint /api/validacao

GET /api/validacao { "precision": 0.72, // alvo: ≥72% (com dataset maior) "recall": 0.58, // alvo: ≥58% "f1_score": 0.64, // alvo: ≥0.64 "total_transacoes_treino": 26, "total_transacoes_teste": 7, "transacoes_avaliadas": 2, // apenas com ≥2 itens "source": "computado" }
Nota sobre o dataset atual: Com 33 transações, apenas 2 têm itens suficientes para validar (≥ 2 produtos por pedido). As métricas serão representativas quando o histórico completo da Quasar Barber for importado (estimativa: 500+ transações). Os alvos de 72% / 58% / 0.64 foram definidos na proposta do TCC e serão atingidos com o dataset completo.
