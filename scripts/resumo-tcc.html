1. Objetivo do Projeto

Desenvolver um sistema de recomendação de produtos B2B para a Quasar Barber, distribuidora de produtos para barbearias em Codó-MA. O sistema analisa o histórico de vendas para identificar padrões de compra e sugerir automaticamente produtos complementares no momento do pedido.

O trabalho aplica o algoritmo de Association Rules (Market Basket Analysis) — técnica consolidada em Data Mining — adaptada ao contexto B2B.

Problema resolvido

  • Vendedores não sabem quais produtos sugerir em cada pedido
  • Oportunidades de cross-sell perdidas por falta de análise de dados
  • Decisões baseadas em intuição, não em histórico quantitativo

Solução proposta

  • API REST: recebe produto → retorna top 3 produtos comprados junto
  • Dashboard React para visualizar padrões e métricas
  • Algoritmo treinado com dados reais de vendas da distribuidora
Exemplo: Vendedor informa "Fox For Men Barba" → sistema retorna "Espuma para Barba Premium (43% confiança)" e "Tônico de Barba (43%)" como sugestões.

6. Algoritmo de Recomendação — Association Rules

As três métricas

MétricaFórmulaSignificado
Supportfreq(A ∩ B) / totalQuão comum é o par A+B. Filtra pares raros.
Confidencefreq(A ∩ B) / freq(A)P(B|A): probabilidade de B dado que A foi comprado.
Liftconfidence(A→B) / P(B)Lift > 1 = correlação real; = 1 = independência.

Thresholds

ParâmetroValorArquivo
MIN_CONFIDENCE0.30 (30%)src/config/constants.js
MIN_SUPPORT0.02 (2%)src/config/constants.js
TOP_N3 recomendaçõessrc/config/constants.js

Fluxo do algoritmo

// Para cada par (A, B) onde A ≠ B: freq_A = vendas onde A aparece freq_AB = vendas onde A E B aparecem confidence = freq_AB / freq_A // P(B|A) support = freq_AB / total // freq. conjunta lift = confidence / P(B) // correlação real? // Aceita regra se confidence >= 0.30 E support >= 0.02 // Ordena por confidence desc → retorna top 3

Exemplo real testado

GET /api/recomendacoes/Fox%20For%20Men%20Barba → Espuma para Barba Premium (confiança: 43%, lift: 1.8) → Tônico de Barba (confiança: 43%, lift: 2.1) Complexidade: O(n × p²) — ~200k ops com 500 transações e 20 produtos Com cache Redis: <10ms (vs ~350ms sem cache)

8. Cache Redis

Estratégia cache-aside

  • 1. Verifica Redis pela chave → se hit, retorna imediatamente
  • 2. Se miss: busca Supabase → calcula algoritmo → armazena no Redis
  • 3. Retorna resultado com campo source: "cache" ou "computado"

Degradação graceful

Se o Redis estiver offline, todas as funções do cache retornam silenciosamente (get → null, set → void) sem lançar erros. A API continua funcionando normalmente, apenas sem o benefício do cache.

~350ms

<10ms
